# Gravity 产品与架构蓝图（内部评审版）

## 1. 蓝图摘要

Gravity 是一个面向企业的“一站式自动化营销运营系统”。它的目标不是替代某一个运营工具，而是把运营工作从“人工执行 + 多工具拼接”升级为“统一数据底座 + 统一决策中枢 + 统一编排执行 + 统一效果回流”的闭环系统。

如果把传统运营系统理解为“工具集合”，Gravity 则是“增长操作系统”：

- 它接入用户行为、线索、渠道回执、订单和客服反馈
- 它根据规则、策略、模型和实验结果自动判断下一步动作
- 它通过工作流和渠道适配器执行动作，并把结果回写
- 它持续学习哪些内容、渠道、节奏和人群组合更有效

本蓝图面向产品、架构、研发和交付团队，用来统一系统边界、能力地图、数据路径、治理原则和评审口径。

## 2. 评审结论先行

### 2.1 我们要解决什么

- 运营动作分散在多个工具里，导致流程断裂、数据割裂、责任不清
- 内容、投放、线索、跟进、成交、复购之间缺少统一的闭环
- 大量重复动作依赖人工执行，成本高、错误率高、难以复制
- 自动化系统通常只覆盖局部，无法形成持续优化的增长系统

### 2.2 Gravity 的核心判断

Gravity 应该被设计成三层统一：

1. **统一数据层**：所有用户、渠道、行为、转化、回执进入同一数据基座
2. **统一决策层**：所有人群、规则、评分、实验、AI 建议进入同一策略中枢
3. **统一执行层**：所有触达、审批、回写、补偿进入同一编排与治理体系

### 2.3 成败标准

如果这个系统最终只做到以下任意一项，它都不算完整：

- 只会发消息
- 只会做工作流
- 只会做 CRM
- 只会做内容生成
- 只会做数据看板

Gravity 只有在“决策、执行、回流、学习、治理”都具备后，才算真正成型。

## 3. 产品愿景

Gravity 的最终形态，是让运营团队从重复劳动中解放出来，把精力放回到目标、策略、创意和异常处理上。

### 3.1 业务愿景

- 让获客、培育、转化、复购、留存、召回和裂变形成统一链路
- 让内容生产、渠道触达、线索流转、销售协同、用户回流形成统一闭环
- 让运营动作尽量自动化，人工仅在策略设定、关键审批和异常处置时介入
- 让每一次动作都能被衡量、被归因、被复盘、被优化

### 3.2 产品定位

Gravity 不是以下几种工具的简单拼接：

- 不是单点营销自动化工具
- 不是纯 CRM
- 不是仅负责发消息的渠道集成平台
- 不是只做工作流编排的低代码工具
- 不是只给运营人员写文案的 AI 工具

Gravity 需要同时具备这些能力，但它们都必须被统一到同一套数据、同一套权限、同一套审计、同一套目标体系中。

## 4. 目标用户与使用场景

### 4.1 目标用户

- **增长负责人**：关注投放效率、线索质量、转化率和 ROI
- **运营负责人**：关注活动编排、内容生产、触达效率和自动化覆盖率
- **销售负责人**：关注线索分配、跟进效率、成交率和漏斗健康度
- **客户成功/留存负责人**：关注复购、活跃、唤醒和流失预警
- **管理者**：关注整体增长、资源投入和风险控制
- **实施/架构团队**：关注集成能力、数据隔离、扩展性和可交付性

### 4.2 典型场景

- 新用户进入后自动进入欢迎与培育链路
- 广告或内容引流后自动完成留资、分配和跟进
- 高意向线索自动升级处理并触发销售动作
- 成交后自动进入复购和会员运营链路
- 沉默用户自动进入召回和再激活链路
- 活动过程中自动 A/B 测试内容和触达节奏
- 某个渠道异常时自动降级、切换或暂停

## 5. 设计原则

### 5.1 闭环优先

任何能力都必须能从输入开始，到执行结束，再把结果回流到下一轮决策。

### 5.2 自动优先

重复度高、规则明确、风险可控的工作优先自动化。

### 5.3 可配置优先

不同业务线、行业和团队差异，通过配置、模板和策略解决，避免硬编码分叉。

### 5.4 可审计优先

系统的每次关键动作都必须能追踪到责任人、来源策略、执行时间和结果。

### 5.5 风险可控

高风险动作保留审批门和治理门，必要时可暂停、回滚、冻结或降级。

### 5.6 多租户优先

同一套产品必须同时支持 SaaS 标准化交付和私有化部署。

## 6. 端到端运营闭环

Gravity 的核心是一个持续运行的闭环：

```text
外部流量 / 用户行为 / 渠道回执 / 业务回流
                │
                ▼
         统一接入与标准化
                │
                ▼
          身份合并与画像更新
                │
                ▼
          分群、评分与策略判断
                │
                ▼
          内容生成与工作流编排
                │
                ▼
          渠道触达与任务执行
                │
                ▼
         行为回传、转化回流、异常告警
                │
                ▼
         归因分析、实验分析、策略回训
                │
                └───────────────► 下一轮动作
```

### 6.1 闭环必须回答的六个问题

1. 现在是谁？
2. 处于什么阶段？
3. 应该对他做什么？
4. 通过什么渠道做？
5. 结果如何？
6. 下一步该怎么优化？

只要其中任何一个问题无法回答，闭环就不完整。

## 7. 产品能力地图

### 7.1 统一数据中心

统一接入所有业务信号，形成可计算的用户与业务基础设施。

- 身份统一：匿名访客、联系人、线索、客户、企业、设备映射
- 行为统一：浏览、点击、留资、加微、咨询、下单、复购、投诉
- 业务统一：活动、内容、渠道、订单、工单、审批、实验
- 分析统一：漏斗、归因、ROI、留存、转化、复购

### 7.2 智能分群与画像

通过规则和模型对人群进行动态切分。

- 标签体系：显式标签、行为标签、推断标签、黑白名单
- 生命周期：新客、潜客、意向、成交、复购、流失、召回
- 目标人群：按产品、渠道、阶段、地域、行为、价值进行组合

### 7.3 决策与策略中枢

系统决定“对谁、在什么时候、通过什么内容、经由哪个渠道、以什么节奏”进行动作。

- 规则引擎：资格判断、频控、合规、预算、黑白名单
- 策略引擎：动作优先级、触达顺序、预算分配、路径选择
- 评分引擎：线索质量、转化概率、流失风险、复购概率
- 实验引擎：A/B 测试、多版本策略、对照组和显著性判断

### 7.4 内容与素材工厂

系统根据业务目标、用户阶段和渠道要求自动产出内容。

- 标题、正文、摘要、海报文案、落地页内容
- 私信、邮件、短信、社群、企微话术
- 多版本素材与实验变体
- 品牌规范校验、敏感词过滤、渠道格式适配

### 7.5 工作流与编排中心

把策略转成具体动作，并在执行过程中管理等待、分支、补偿和重试。

- 事件触发
- 条件分支
- 延迟等待
- 并行执行
- 人工审批
- 失败补偿
- 结果回写

### 7.6 渠道执行网络

统一管理真实世界中的触达与回执。

- 邮件、短信、微信、企微、内容平台、广告平台、落地页、表单、客服
- 所有渠道统一通过适配器接入
- 统一接收发送结果、打开、点击、回复、退订、投诉、转化事件

### 7.7 归因与学习系统

系统必须知道“为什么有效”，而不仅仅是“有没有效果”。

- 漏斗分析：每个阶段的转化损耗
- 归因分析：不同触点对结果的贡献
- 实验分析：不同策略的增益对比
- 策略回训：把高胜率组合沉淀为默认策略

### 7.8 风险治理中心

自动化越强，治理越重要。

- 审批流
- 审计日志
- 配额与预算控制
- 频控与投诉保护
- 渠道降级与回滚
- 多租户隔离与权限控制

## 8. 架构蓝图

### 8.1 分层结构

Gravity 采用“模块化单体 + 事件驱动”的架构方式。

- **接入层**：接收外部请求、Webhook、回调和批量导入
- **数据层**：保存业务数据、状态数据和分析数据
- **画像层**：统一身份、标签、属性和生命周期
- **决策层**：规则、评分、策略、实验、AI 建议
- **编排层**：工作流、调度、补偿、恢复
- **执行层**：渠道发送、任务执行、状态回写
- **分析层**：漏斗、归因、ROI、实验、看板
- **治理层**：权限、审计、审批、风控、降级

### 8.3 关键设计选择

- 事务与分析分离：OLTP 使用 PostgreSQL，分析使用 ClickHouse
- 状态与事件分离：流程状态与行为事件不混表
- 定义与执行分离：工作流定义与执行实例分离，便于版本化
- 决策与执行分离：策略先生成 `ActionRequest`，再由执行层落地
- 渠道与业务分离：渠道通过适配器接入，不侵入领域逻辑

### 8.4 推荐技术承载

- 前端：React + TypeScript + Vite
- 后端：Rust + Axum
- 事务存储：PostgreSQL
- 分析存储：ClickHouse
- 缓存与短期状态：Redis
- 消息与编排：NATS
- 部署：Docker Compose / Kubernetes

## 9. 关键对象模型

以下对象构成系统的业务骨架：

- **Organization**：租户和组织边界
- **User**：系统操作者和权限主体
- **Identity**：跨渠道身份映射
- **Contact / Lead**：联系人和线索统一模型
- **Segment**：动态人群定义
- **Journey**：生命周期状态机
- **Campaign**：活动与目标配置
- **Workflow**：自动化流程定义
- **Action**：系统发起的动作
- **Task**：待执行或待跟进任务
- **ContentAsset**：内容、模板和素材
- **ChannelAccount**：渠道账号和授权信息
- **Event**：用户行为、渠道回执、系统事件
- **Conversion**：转化结果和价值记录
- **Experiment**：实验配置和结果
- **Approval**：审批记录
- **AuditLog**：审计记录

## 10. 关键业务流程

### 10.1 内容到留资

1. 系统选择目标人群
2. 系统生成或选取内容版本
3. 系统发布到渠道或落地页
4. 用户留资后进入统一画像
5. 系统自动打标签、分群、分配或进入后续流程

### 10.2 线索到成交

1. 系统对线索进行评分和排序
2. 系统按规则分配给销售或自动培育
3. 系统跟踪跟进、回复和转化动作
4. 系统自动触发催单、提醒或升级处理
5. 成交结果回流到策略与分析系统

### 10.3 成交到复购

1. 系统识别购买后阶段
2. 系统启动会员、交叉销售或续费流程
3. 系统自动识别复购机会和流失风险
4. 系统触发个性化召回或升级触达

### 10.4 召回与留存

1. 系统检测沉默或流失风险
2. 系统选择召回策略与渠道
3. 系统执行低风险自动触达
4. 系统在高风险动作上走审批和风控

## 11. 时序视图

### 11.1 新线索进入后的标准链路

```text
线索进入
  -> 标准化与去重
  -> 身份合并
  -> 标签/画像更新
  -> 评分与分群
  -> 选择策略和工作流
  -> 生成内容
  -> 渠道执行
  -> 用户响应回流
  -> 结果归因
  -> 下一步动作
```

### 11.2 高风险动作审批链路

```text
策略命中高风险规则
  -> 创建审批单
  -> 通知审批人
  -> 审批通过 / 拒绝
  -> 通过后进入执行层
  -> 拒绝后进入安全回退
  -> 写审计日志
```

### 11.3 渠道失败降级链路

```text
发送失败 / 回执超时 / 限流
  -> 记录失败原因
  -> 尝试重试或切换备用渠道
  -> 若仍失败则降级
  -> 必要时暂停工作流
  -> 告警并等待人工介入
```

## 12. 数据与事件设计

Gravity 的核心数据不是“表”，而是“可追踪事件 + 可恢复状态 + 可分析结果”的组合。

### 12.1 三类数据

- **主业务数据**：组织、联系人、活动、工作流、渠道账号、审批
- **过程状态数据**：工作流实例、任务状态、调度信息、锁、频控
- **分析数据**：行为事件、漏斗聚合、归因结果、实验结果

### 12.2 关键事件

- 用户行为事件：浏览、点击、提交、加微、下单、复购
- 渠道回执事件：发送成功、送达、打开、点击、退订、投诉
- 系统事件：策略生成、工作流启动、审批通过、执行失败、告警

### 12.3 数据原则

- 事件优先追加写入
- 状态更新必须可幂等
- 工作流定义必须版本化
- 审计日志必须不可抵赖
- 分析计算不得反向污染事务核心

## 13. 多租户与安全治理

### 13.1 多租户

- 默认以租户为核心隔离边界
- 所有业务数据必须可按租户过滤
- SaaS 模式下支持共享服务与隔离数据层
- 私有化模式下支持独立部署与独立资源

### 13.2 权限与审计

- 基于 RBAC 控制组织、角色和动作权限
- 高风险动作必须审批
- 每次关键写操作都必须生成审计记录
- 管理员、运营、销售、审计等角色权限必须分离

### 13.3 风险边界

- 大额预算必须受控
- 批量触达必须受控
- 敏感内容必须校验
- 外部凭证必须加密
- 渠道失败必须可降级

## 14. 失败模式与处理策略

### 14.1 常见失败模式

- 触发重复导致重复执行
- 渠道限流导致消息未送达
- 数据回流延迟导致画像不准
- 审批链路阻塞导致动作停滞
- 模型不可用导致策略退化
- 多租户隔离配置错误导致越权风险

### 14.2 处理策略

- 幂等键 + 去重表 + 执行锁
- 失败重试 + 指数退避 + 备用渠道
- 延迟补偿 + 再计算 + 数据回放
- 审批超时升级 + 兜底路径
- 规则引擎回退 + 手工干预
- 权限测试 + 审计追踪 + 隔离校验

## 15. 成功标准

Gravity 的成功不只是“能发消息”，而是以下指标显著提升：

- 运营动作自动化覆盖率
- 线索响应速度
- 触达效率和到达率
- 留资率和成交率
- 复购率和留存率
- ROI 和 LTV
- 人工介入率下降
- 风险事件可控率

## 16. 演进路径

### 16.1 第一阶段

打通内容到留资、线索到成交的核心链路。

### 16.2 第二阶段

补齐成交到复购、留存与召回的自动化能力。

### 16.3 第三阶段

强化实验、归因、策略回训与自治优化能力。

### 16.4 最终阶段

形成“人设目标、系统执行、自动优化”的增长操作系统。

## 17. 术语表

- **线索（Lead）**：已经表达过兴趣或留下可跟进信息的潜在客户
- **联系人（Contact）**：系统中统一管理的用户或客户实体
- **画像（Profile）**：联系人属性、标签、阶段和推断结果的集合
- **分群（Segment）**：按规则或模型动态组成的人群集合
- **工作流（Workflow）**：按触发条件编排的自动化执行流程
- **动作（Action）**：系统希望执行的具体运营行为
- **任务（Task）**：需要完成或跟踪的执行单元
- **回执（Receipt）**：渠道返回的发送、打开、点击、退订等结果
- **归因（Attribution）**：将结果分配给不同触点或策略的过程
- **实验（Experiment）**：用于比较不同策略效果的对照机制
- **审批（Approval）**：对高风险动作进行人工放行的治理机制
- **审计（Audit）**：对关键操作进行记录和追踪的机制

## 18. 内部评审关注点

### 18.1 架构评审

- 是否真的需要模块化单体 + 事件驱动，而不是纯微服务
- 决策层、编排层、执行层是否边界清晰
- 事务数据与分析数据是否真正解耦

### 18.2 产品评审

- 是否足以覆盖增长、运营、销售、留存四类核心场景
- 是否能让运营动作从手工变成闭环自动化
- 是否能让用户理解系统为什么做出某个动作

### 18.3 治理评审

- 高风险动作是否都能被拦截和审计
- 多租户隔离是否足够强
- 外部渠道凭证和敏感数据是否安全

### 18.4 交付评审

- SaaS 与私有化的能力是否共享同一套核心逻辑
- 是否能在不改核心逻辑的情况下接入新渠道
- 是否能在渠道故障时维持系统可用性

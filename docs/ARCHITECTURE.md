# Gravity 架构设计

## 1. 架构目标

Gravity 的架构目标不是“做一个营销工具”，而是构建一个能够持续自动运行的运营系统。系统需要同时满足以下要求：

- 全渠道接入，统一承接外部流量、私域触点和用户行为信号
- 全链路编排，把获客、培育、转化、复购、留存、召回和裂变串成闭环
- 自动化决策，能够基于规则、策略、模型和实验结果自动选择动作
- 自动化执行，能够自动下发任务、触达用户、更新状态并回传结果
- 可控可审计，对高风险动作、敏感内容和关键预算保留审批和审计
- 双模式交付，同一套核心架构同时支持 SaaS 与私有化部署

## 2. 总体架构

Gravity 采用**模块化单体 (Modular Monolith)** 作为核心实现形态，外部通过统一 API 和事件总线连接渠道、前端和分析系统，内部按领域边界拆分模块。

## 3. 核心分层

### 3.1 数据层

负责统一采集和管理组织、用户、联系人、事件、活动、任务、内容、渠道账号、实验和审计数据。

- PostgreSQL：主业务数据、事务、配置、权限、多租户隔离
- Redis：缓存、分布式锁、会话、频控、短时状态
- ClickHouse：行为事件、漏斗分析、归因分析、ROI 和时序聚合

### 3.2 身份与画像层

负责统一用户身份、线索归并、标签、属性、生命周期状态和分群结果。

- 多渠道身份合并
- 标签体系与动态分群
- 生命周期阶段识别
- 用户状态机管理

### 3.3 策略与决策层

负责决定“对谁做什么、什么时候做、通过什么渠道做、是否需要审批”。

- 规则引擎：频控、黑白名单、资格判断、兜底逻辑
- 策略引擎：用户分层、动作优先级、触达选择、预算分配
- 实验引擎：A/B 测试、多版本策略、对照组管理
- 预测能力：发送时机、转化倾向、流失风险、复购概率

### 3.4 内容与 AI 层

负责生产和改写面向不同渠道、不同人群、不同阶段的内容资产。

- 文案生成：标题、正文、私信、邮件、短信、社群话术
- 素材生成：海报、落地页内容、活动说明、脚本和摘要
- 渠道改写：自动适配不同平台的格式、长度和语气
- 代理协作：由 AI 代理完成归纳、草拟、变体生成和复盘摘要

### 3.5 执行与编排层

负责把策略转化为可执行任务，并保证可靠性、可恢复性和幂等性。

- 工作流编排：触发、分支、等待、升级、补偿、重试
- 任务调度：定时任务、延迟任务、队列任务、失败重放
- 触达执行：邮件、微信、企微、短信、社群、广告平台、落地页
- 状态回写：任务结果、用户反馈、事件回传、状态更新

### 3.6 分析与学习层

负责将执行结果反馈给系统，让系统持续优化。

- 漏斗分析：曝光、点击、回复、加微、预约、成交、复购
- 归因分析：首触点、末触点、线性、规则型归因
- 结果学习：成功策略沉淀、失败策略降权、自动推荐优化动作
- 运营看板：渠道健康、内容胜率、自动化覆盖率、人工介入率

### 3.7 风控与治理层

负责控制系统在自动化运行过程中的风险。

- 高风险动作审批：大额投放、敏感话术、批量触达、关键配置变更
- 安全控制：凭证加密、权限隔离、审计日志、操作追踪
- 频控与投诉保护：触达频率、黑名单、退订、投诉和合规处理
- 降级策略：渠道失败、数据缺失、模型不可用时自动回退到规则模式

## 4. 关键领域对象

- **Organization**：租户和组织边界
- **User**：系统操作者和权限主体
- **Contact / Lead**：联系人和线索统一模型
- **Segment**：分群与动态人群包
- **Campaign**：营销/运营活动载体
- **Workflow**：自动化流程定义
- **Task**：系统下发的执行任务
- **ContentAsset**：文案、图片、脚本和模板
- **ChannelAccount**：渠道账号和授权信息
- **Event**：用户行为、渠道回执、系统事件
- **Conversion**：转化结果和价值记录
- **Experiment**：实验配置和对照结果
- **Approval**：审批记录和风险放行
- **AuditLog**：操作审计和责任追踪

## 5. 事件流与闭环

Gravity 的核心不是单次触达，而是持续闭环。

```
用户行为 / 外部回调 / 系统事件
        │
        ▼
  Event Ingestion
        │
        ├──► 画像更新
        ├──► 策略重新计算
        ├──► 工作流触发
        ├──► 分析入库
        └──► 实验结果归档
        │
        ▼
  下一轮自动动作生成
```

系统必须保证每一轮动作都能产生可观察结果，并把结果反向喂给策略层和内容层，从而不断提升命中率、转化率和效率。

## 6. 工作流模型

工作流是 Gravity 的执行骨架，所有运营动作都应抽象成可编排的流程节点。

### 支持的节点类型

- 触发器：事件触发、定时触发、条件触发、手动触发
- 执行动作：发送消息、写入标签、更新属性、创建任务、调用 webhook
- 控制节点：条件分支、等待、并行、合并、重试、补偿
- 智能节点：内容生成、策略推荐、意图识别、风险判断
- 治理节点：审批、降级、审计、告警

### 工作流要求

- 支持幂等执行
- 支持断点恢复
- 支持失败补偿
- 支持按人群和事件动态触发
- 支持人工审批插入到关键节点

## 7. 渠道集成原则

所有渠道统一通过 `ChannelAdapter` 抽象接入，保证触达执行、状态回传和指标采集的一致性。

- 邮件：模板化发送、打开/点击回传、退订处理
- 微信/企微：消息触达、标签同步、会话回传
- 短信：高优先级通知、验证和催付
- 社交内容平台：内容发布、互动回收、引流归因
- 广告平台：投放同步、消耗回传、转化对账
- 落地页/表单：转化承接、表单提交、事件埋点

## 8. 权限与多租户

- 默认使用 PostgreSQL RLS 做租户隔离
- 使用 RBAC 控制组织、团队、角色和动作权限
- 高风险接口需要审批和审计
- 所有外部渠道凭证必须加密存储并支持轮换

## 9. 部署形态

### SaaS

- 适合标准化交付和快速开通
- 支持租户隔离、计费、套餐和配额控制
- 适合统一升级和集中运维

### 私有化

- 适合数据敏感、合规要求高或深度定制的企业
- 可部署在客户自有云、专有云或内网环境
- 支持独立数据库、独立队列和独立网关

## 10. 技术模块映射

| crate | 职责 |
|------|------|
| `gravity-api` | API 网关、认证、租户、权限、编排入口 |
| `gravity-core` | 领域模型、业务规则、状态定义 |
| `gravity-db` | 数据访问、迁移、仓储实现 |
| `gravity-channels` | 渠道适配、凭证管理、触达执行 |
| `gravity-workflow` | 流程引擎、调度、恢复、补偿 |
| `gravity-analytics` | 事件分析、漏斗、归因、报表 |
| `gravity-common` | 公共工具、ID、配置、加密、时间 |

## 11. 关键接口边界

- API 层只做鉴权、校验、路由和编排，不承载业务规则
- Core 层承载领域规则，避免直接依赖外部 IO
- Channel 层负责外部平台交互和回执标准化
- Workflow 层负责执行顺序和恢复逻辑
- Analytics 层负责数据聚合和指标计算

## 12. 设计约束

- 避免把人工操作设计成主要路径，人工只处理策略、审批和异常
- 避免单渠道耦合，所有能力都应支持多渠道扩展
- 避免“只看转化不看反馈”，所有动作都要能回流数据
- 避免把 AI 当作唯一决策来源，AI 应与规则和风控并存

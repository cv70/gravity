# Gravity - 一站式自动化营销运营增长转化平台

> 面向企业的全自动运营中台，统一承接获客、培育、转化、复购、留存、召回和裂变等全链路运营工作。

## 项目定位

Gravity 不是单点营销工具，也不是单纯的活动管理系统，而是一个可持续运行的自动化运营系统。系统通过数据采集、用户识别、智能分群、策略决策、内容生成、渠道执行、效果回传和持续优化，把原本依赖人工的运营工作尽可能交给系统自动完成。

## 最终目标

- 自动化掉日常运营中的重复劳动、流程协同和手工执行
- 让运营从“人盯人、人催人、人整理数据”升级为“系统驱动、规则驱动、模型驱动”
- 覆盖从获客到成交、从成交到复购、从流失到召回的完整闭环
- 通过持续学习和实验优化，让系统越跑越准、越跑越高效
- 以少量人工审批，支撑尽量无人值守的高效率运营

## 核心能力

- **统一客户与线索中心** - 汇聚多渠道用户数据，建立统一身份、标签、分群和行为画像
- **自动化工作流引擎** - 触发、分支、等待、升级、回退、重试、补偿全部可编排
- **内容与话术引擎** - 自动生成和改写文案、私信、邮件、短信、社群话术和活动素材
- **全渠道触达执行** - 统一调度微信、企微、邮件、短信、社群、短视频、广告和落地页
- **智能转化编排** - 自动判断触达时机、内容版本、下一步动作和转化路径
- **复购与留存自动化** - 自动召回沉默用户、唤醒老客、推动复购和会员运营
- **归因与实验系统** - 支持漏斗、归因、A/B 测试、策略对照和效果分析
- **风控与审批治理** - 对高风险动作保留审批，对低风险动作默认自动执行
- **企业级权限与审计** - 支持组织、角色、审计日志、多租户隔离和安全控制

## 系统原则

1. **全链路闭环**：每一次触达都必须形成反馈，反馈反向驱动下一次动作。
2. **自动优先**：能自动化的运营动作尽量自动化，人工只做策略、审批和异常处理。
3. **统一编排**：所有渠道、任务、素材、实验和回传都应进入统一编排体系。
4. **策略可配置**：不同业务、不同阶段、不同用户群可以用配置和模板快速适配。
5. **结果可衡量**：所有动作都要能追踪、归因、复盘和优化。

## 技术栈

| 层级 | 技术 |
|------|------|
| 前端 | TypeScript + React + Vite |
| 后端 | Rust (Axum) |
| 数据库 | PostgreSQL + ClickHouse + Redis |
| 消息队列 | NATS |
| 部署 | Docker Compose / Kubernetes |

## 运行形态

- **企业级 SaaS**：适合标准化交付、多租户管理和快速上线
- **私有化部署**：适合对数据安全、网络边界和定制化要求更高的企业
- **双模式兼容**：同一套核心能力支持 SaaS 与私有化两种交付方式

## 项目结构

```
gravity/
├── crates/               # Rust 后端 (模块化单体)
│   ├── gravity-api/      # HTTP API 层
│   ├── gravity-core/     # 核心领域模型
│   ├── gravity-db/       # 数据库访问层
│   ├── gravity-channels/ # 第三方渠道集成
│   ├── gravity-workflow/ # 工作流引擎
│   ├── gravity-analytics/# 分析引擎
│   └── gravity-common/   # 共享工具
├── frontend/             # React 前端
├── docker-compose.yml    # 本地开发环境
└── Cargo.toml            # Rust workspace
```

## 快速开始

```bash
# 启动本地开发环境
docker compose up -d

# 后端开发
cd crates/gravity-api && cargo run

# 前端开发
cd frontend && npm install && npm run dev
```
